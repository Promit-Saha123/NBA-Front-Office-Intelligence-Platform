# Scenario Engine Specification

## 1. Purpose

The scenario engine estimates how a hypothetical roster change could affect a team’s projected performance.

The engine combines:

* versioned player-impact projections
* transparent rotation assumptions
* deterministic team aggregation
* a documented win-conversion method
* descriptive team-profile changes
* grounded explanation factors

The scenario engine is the main differentiating feature of the project.

---

## 2. Product Claim

The scenario engine provides an assumption-driven estimate.

It does not claim to predict:

* the coach’s exact rotation
* exact player roles
* exact season wins
* chemistry
* leadership
* injuries
* playoff outcomes
* tactical adjustments
* guaranteed real-world results

The product should state:

> Scenario results combine model-generated player projections with documented rotation assumptions and deterministic calculations.

As a historical product (decision 0005), the interface must also visibly disclose:
the supported historical season range, data source and license, data version,
model version, minutes-method version, win-conversion version, historical-only
status, that scenario outputs are estimates, and that no current-season claim is
made.

---

## 3. Epistemic Categories

Every output should be classified as one of the following:

### Model prediction

Generated by a trained player-impact model (future PCE phase only — the free MVP
ships no trained model).

Example:

```text
Projected next-season PCE
```

### Historical benchmark / synthetic estimate (free MVP — decision 0007)

Contribution values in the free MVP are either **historical RAPTOR benchmark**
values (FiveThirtyEight, CC BY 4.0, attributed) or **synthetic contribution
estimates** (always labeled). They are not model predictions, not PCE, and not
causal; use the exact labels in decision 0007 §8.

Note on the Player Contribution Estimate (PCE) itself
([decision 0003](decisions/0003-internal-player-impact-target.md)): a PCE value is
a statistically estimated metric; applying frozen, versioned PCE coefficients to
canonical player statistics is a **deterministic calculation**; a next-season PCE
forecast is a **model prediction**; team and roster aggregation from fixed
projections and minutes is a **deterministic calculation**. Standard phrase:
*"Statistically estimated metric calculated deterministically from fixed,
versioned coefficients."* Never present PCE as causal, official, definitive, or
equivalent to BPM, RAPTOR, EPM, or another established metric.

### Heuristic assumption

Generated by a documented rule-based method.

Example:

```text
Default player minutes
```

### Deterministic calculation

Directly computed from model outputs, data, or user inputs.

Example:

```text
Minutes-weighted impact change
```

### Descriptive interpretation

A plain-language description of calculated profile differences.

Example:

```text
Three-point volume increased while defensive rebounding declined.
```

The API and interface must preserve these distinctions.

---

## 4. Initial Scenario Scope

The scenario engine operates on **historical seasons only**
([decision 0005](decisions/0005-historical-only-product-scope.md)). The first
scenario implementation supports:

* one selected historical season
* one team from that season
* one player removed
* one player added — must have a player-season record in the same season and not
  already be on the selected roster (no cross-era swaps)
* one baseline roster
* one default rotation
* one scenario rotation
* one projected team-impact comparison
* one projected win-change estimate

Multi-player transactions should be added only after the one-for-one flow is stable.

---

## 5. Core Scenario Flow

```text
Load baseline roster
→ validate requested roster change
→ remove outgoing player
→ add incoming player
→ generate default scenario minutes
→ validate rotation
→ load player projections
→ calculate minutes-weighted team impact
→ calculate team-rating change
→ estimate projected win change
→ calculate descriptive profile changes
→ generate grounded explanation factors
→ return assumptions and versions
```

## 6. Scenario Request Contract

Initial request:

```json
{
  "team_id": 1610612747,
  "season": "2014-15",
  "player_added_id": 12345,
  "player_removed_id": 67890
}
```

Optional future fields:

```text
custom_minutes
rotation_preferences
scenario_name
```

Custom minutes are not required for the first automatic scenario implementation.

## 7. Scenario Response Contract

A scenario response should include:

```json
{
  "team_id": 1610612747,
  "season": "2014-15",
  "baseline": {
    "projected_wins": 35.2,
    "projected_team_impact": -1.8
  },
  "scenario": {
    "projected_wins": 38.6,
    "projected_team_impact": -0.4
  },
  "changes": {
    "projected_wins": 3.4,
    "projected_team_impact": 1.4
  },
  "rotation": [],
  "profile_changes": {},
  "explanation_factors": [],
  "contribution_provider": "historical-raptor-benchmark-v1",
  "contribution_epistemic_type": "historical_benchmark",
  "model_version": null,
  "data_version": "fivethirtyeight-nba-raptor-2022-11-29",
  "minutes_method": "heuristic-v1",
  "minutes_assumptions": {
    "editable": false,
    "validated": false
  },
  "win_conversion_version": "net-rating-to-wins-v1"
}
```

The response should avoid unsupported precision.

Values may be rounded for display while preserving full precision internally.

## 8. Roster Validation

Before calculation, validate:

* team exists
* season is supported
* outgoing player is on the baseline roster
* incoming player is not already on the roster
* incoming and outgoing players differ
* both players have usable identities
* required projections exist
* roster size remains valid
* duplicate players do not exist
* scenario does not produce impossible state

Use stable machine-readable errors.

Examples:

```text
TEAM_NOT_FOUND
PLAYER_NOT_FOUND
PLAYER_NOT_ON_ROSTER
PLAYER_ALREADY_ON_ROSTER
MISSING_PLAYER_PROJECTION
INVALID_ROSTER_SCENARIO
```

---

## 9. Baseline Roster

The baseline roster is the selected team's **historical roster for the selected
season**, from a versioned historical-roster snapshot derived from the approved
historical data.

Each roster record should include:

```text
internal_player_id
team_id
season
roster_status
source
data_version
```

The engine should not fetch a live roster during each scenario request.

Roster data should be ingested and validated separately.

---

## 10. Baseline Minutes

Baseline minutes are a heuristic unless explicitly sourced from an approved projection system.

Potential inputs include:

* recent minutes per game
* prior-season minutes
* starter status
* games played
* recent role
* position eligibility
* roster competition

Do not hardcode apparently authoritative weighting values without documentation.

All configurable values should live in a named configuration object.

Example:

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class MinutesAllocationConfig:
    total_team_minutes: float = 240.0
    max_player_minutes: float = 40.0
    minimum_rotation_minutes: float = 6.0
    maximum_rotation_size: int = 10
```

Any unvalidated method must remain labeled as heuristic.

---

## 11. Scenario Minutes Allocation

The v1 allocation method should be simple and inspectable.

Suggested process:

### Step 1: Start from baseline minutes

Load the baseline rotation and normalize it to 240 minutes if needed.

### Step 2: Remove outgoing player

Set the outgoing player’s scenario minutes to zero.

Add those minutes to the available pool.

### Step 3: Add incoming player

Assign the incoming player a provisional role based on available evidence, such as:

* recent minutes
* starter status
* projected role
* position
* quality tier

### Step 4: Redistribute remaining minutes

Redistribute remaining minutes among eligible roster players based on documented rules.

### Step 5: Enforce constraints

Normalize and repair the rotation until it satisfies all hard constraints.

Every repair must be visible in logs or returned metadata.

---

## 12. Rotation Constraints

The engine must enforce:

* total minutes equal exactly 240
* no negative minutes
* no player exceeds configured maximum
* removed player receives zero minutes
* only rostered players receive minutes
* no duplicate player rows
* rotation size remains within configured bounds
* players below minimum rotation minutes are either removed from the active rotation or explicitly retained
* basic positional viability

Floating-point comparison should use a documented tolerance internally, but stored and displayed rotation totals should resolve to 240.

---

## 13. Positional Viability

The first version should use a simple, transparent position system.

Possible normalized positions:

```text
guard
wing
forward
center
```

or:

```text
point guard
shooting guard
small forward
power forward
center
```

The project must document how multi-position players are represented.

Basic viability may require sufficient minutes from:

* primary ball handlers
* perimeter players
* forwards
* interior players

Do not present positional viability as a sophisticated tactical model.

It is a constraint system intended to prevent obviously invalid rotations.

---

## 14. Minutes Method Metadata

Every scenario should identify the minutes method.

Example:

```json
{
  "minutes_method": "heuristic-v1",
  "minutes_assumptions": {
    "validated": false,
    "editable": false,
    "total_minutes": 240,
    "maximum_player_minutes": 40
  }
}
```

A future validated method should receive a new version.

Never reuse `heuristic-v1` after materially changing its behavior.

---

## 15. Editable Minutes

Editable minutes belong to the scenario-sensitivity phase.

The user should be able to:

* adjust player minutes
* see the current total
* receive validation feedback
* reset to defaults
* rerun the scenario
* compare default and edited outputs

The UI must not silently rebalance user-entered minutes unless that behavior is clearly disclosed.

Possible behavior:

```text
User enters invalid rotation
→ show validation error
→ do not calculate until corrected
```

or:

```text
User requests auto-balance
→ engine adjusts values
→ show every adjustment
```

---

## 16. Player Contribution Inputs (provider abstraction — decision 0007)

The scenario engine loads player contribution values only through the
contribution-provider interface; it must not depend on RAPTOR-specific (or any
provider-specific) field names.

```text
ContributionProvider (interface)
- get_player_contribution(player_id, season)
- get_provider_version()
- get_data_version()
- get_epistemic_type()
```

Implementations: `HistoricalRaptorBenchmarkProvider` and
`SyntheticContributionProvider` in the free MVP; `PceProvider` in the future PCE
phase (whose responses will also carry `model_version` and
`feature_schema_version` per [ml-specification.md §20](ml-specification.md)).
Every scenario response must expose the provider type, provider version, and data
version, and the UI must render the matching label from decision 0007 §8
(benchmark values are never presented as PCE or a house metric).

A missing contribution value must produce an explicit error or approved fallback.

Do not silently substitute league-average impact without disclosure.

Any fallback method must have:

* a version
* a documented rationale
* explicit response metadata
* tests

---

## 17. Team Impact Aggregation

The initial team-impact estimate should use a documented minutes-weighted aggregation.

Conceptually:

```text
player projected impact
×
share of team minutes
→
team projected impact contribution
```

The exact formula must be documented.

Do not treat a player projected for 12 minutes the same as a player projected for 36 minutes.

The baseline and scenario must use the same aggregation method.

---

## 18. Replacement-Level and Bench Treatment

The engine must define how it handles:

* players without projections
* end-of-bench players
* two-way players
* replacement players
* unfilled roster spots

Potential approaches include:

* explicit replacement-level projection
* exclusion from active rotation
* conservative documented fallback

The chosen approach must not be hidden.

---

## 19. Team Rating Conversion

The scenario engine should convert aggregated player impact into a team-level rating using a documented versioned method.

Conceptual flow:

```text
minutes-weighted player impact
→ projected team rating
→ projected net rating
```

The first method may be calibrated against historical team-season data.

It must document:

* input metric
* formula or fitted relationship
* training period, where applicable
* assumptions
* evaluation error
* version

Do not claim that PCE sums directly equal team net rating unless this relationship has been validated for the frozen PCE version.

---

## 20. Win Conversion

Projected wins should be derived from projected team rating or net rating using a documented method.

Potential methods:

* historically fitted linear relationship
* Pythagorean expectation
* logistic or nonlinear historical mapping

The selected method must be:

* versioned
* tested
* reproducible
* evaluated historically
* applied identically to baseline and scenario

Example metadata:

```text
win_conversion_version = net-rating-to-wins-v1
```

Do not hardcode unexplained conversions.

---

## 21. Fit and Team Profiles

In v1, fit outputs are descriptive.

Possible profile categories:

* shooting
* three-point volume
* spacing proxy
* scoring
* playmaking
* turnover control
* offensive rebounding
* defensive rebounding
* steal activity
* block activity
* availability
* positional balance

Profile values may be calculated using:

* minutes-weighted player rates
* standardized league-relative scores
* percentile ranks
* normalized team indices

The method must be documented.

---

## 22. No Arbitrary Win Adjustments

Do not apply hand-tuned fit bonuses or penalties to projected wins.

Prohibited examples:

```text
+0.8 wins for spacing
-0.5 wins for role overlap
+1.0 wins for positional balance
```

Such effects may only enter the win calculation after:

* a historical methodology is defined
* coefficients or relationships are estimated
* out-of-sample evaluation is completed
* a decision record is approved
* a new scenario-engine version is created

Until then, use the label:

> Heuristic scenario profile, not a validated causal fit model.

---

## 23. Explanation Generation

Explanation factors must come from calculated differences.

Example factor structure:

```json
{
  "metric": "three_point_volume",
  "baseline_value": 34.2,
  "scenario_value": 38.5,
  "change": 4.3,
  "direction": "increase",
  "importance": 0.82
}
```

The interface may turn this into:

> Three-point volume increased meaningfully.

Do not generate claims unsupported by a factor.

Prohibited claims include:

* leadership
* mentality
* clutch performance
* chemistry
* winning culture
* locker-room impact

unless the system has a documented source and methodology for them.

---

## 24. LLM Use

An LLM is not required for v1.

A future LLM explanation layer may:

* summarize verified factors
* improve readability
* answer questions using stored scenario outputs

It may not:

* invent new metrics
* create unsupported basketball reasoning
* alter numerical outputs
* suppress limitations
* imply causal certainty

The deterministic explanation factors must remain stored and inspectable.

---

## 25. Sensitivity Analysis

The sensitivity phase should show how results change when assumptions change.

Potential controls:

* incoming player minutes
* outgoing player replacement minutes
* maximum-player workload
* active rotation size
* replacement-level assumption

The product should display:

* default result
* edited result
* difference caused by assumption changes
* whether model projections changed
* whether only heuristic assumptions changed

This reinforces the distinction between model prediction and scenario assumption.

---

## 26. Determinism

For a fixed combination of:

* baseline roster
* roster change
* player projections
* model version
* data version
* minutes method
* configuration
* win-conversion version

the scenario result must be deterministic.

Persist or expose these versions in every response.

Randomness is not needed in the first scenario engine.

If simulation is introduced later, random seeds and distributions must be recorded.

---

## 27. Persistence

The initial engine may calculate scenarios without saving them.

When scenario persistence is added, store:

```text
scenario_id
team_id
season
baseline_roster_version
players_added
players_removed
rotation
model_version
data_version
minutes_method
minutes_configuration
win_conversion_version
result
created_at
```

Saved scenarios must remain reproducible after newer models are deployed.

Do not recalculate historical saved scenarios silently using new model versions.

---

## 28. Caching

Scenario results may be cached when the full input signature matches.

Cache keys should include:

* team
* season
* added player
* removed player
* custom minutes
* model version
* data version
* minutes method
* configuration version
* win-conversion version

Do not return cached results generated under incompatible versions.

Caching is an optimization, not a requirement for the first loop.

---

## 29. Performance Targets

Initial targets:

* baseline roster load under 500 milliseconds locally
* scenario calculation under 2 seconds locally
* model artifact loaded once at application startup
* no N+1 database access
* no live external request during a scenario calculation

These are engineering targets, not production guarantees.

---

## 30. Logging and Observability

Scenario logs should include:

* request identifier
* scenario identifier
* team
* season
* outgoing player
* incoming player
* model version
* data version
* minutes method
* win-conversion version
* execution duration
* validation repairs
* warnings
* error category

Do not log secrets or unnecessarily large player records.

Track operational metrics such as:

* scenario request count
* scenario failure count
* average calculation duration
* missing-projection count
* invalid-rotation count
* cache hit rate, when caching exists

---

## 31. Error Handling

Expected errors include:

* invalid team
* invalid season
* invalid incoming player
* invalid outgoing player
* outgoing player not on roster
* incoming player already on roster
* missing player projection
* incompatible model version
* invalid rotation
* unavailable baseline roster
* failed team-rating conversion
* failed win conversion

Return consistent machine-readable codes and clear user-facing messages.

Do not expose internal stack traces.

---

## 32. Testing Requirements

Add tests for:

### Roster changes

* valid one-for-one swap
* outgoing player not on roster
* incoming player already on roster
* same player added and removed
* duplicate roster entries

### Minutes

* total equals 240
* no negative minutes
* maximum minutes enforced
* removed player receives zero
* incoming player receives expected provisional role
* deterministic allocation
* invalid position coverage
* repair behavior is reported

### Projection loading

* all rotation players have projections
* missing projection returns explicit error
* incompatible season fails
* model and data versions are returned

### Aggregation

* baseline and scenario use identical formulas
* minutes weighting is correct
* known synthetic examples produce expected output
* replacement-level handling is correct

### Win conversion

* known net rating produces expected wins
* version is returned
* invalid input fails clearly
* baseline and scenario use the same conversion

### Explanations

* every explanation maps to a calculated factor
* no unsupported factors appear
* direction matches numerical change
* zero-change factors are suppressed where appropriate

### Determinism

* identical requests produce identical results
* version changes invalidate cached results
* configuration changes alter the signature

---

## 33. Dependency Rules

Do not add an optimization, simulation, or rules-engine library unless the existing implementation cannot reasonably support the requirement.

Before adding a dependency, document:

* why it is needed
* alternatives considered
* maintenance status
* licensing
* effect on deployment
* effect on determinism
* effect on reproducibility

The initial minutes engine should be implementable with normal application code.

---

## 34. Definition of Done

A scenario-engine change is complete only when:

* the roster change is validated
* the rotation totals exactly 240 minutes
* assumptions are exposed
* model and data versions are exposed
* the minutes method is exposed
* the win-conversion version is exposed
* results are deterministic
* explanations map to calculated values
* no arbitrary fit constant was introduced
* relevant tests pass
* logs are sufficient to diagnose failures
* documentation is updated
* limitations are stated honestly
* the final diff is reviewed

---

## 35. First Deliverable

The first credible scenario deliverable is:

```text
Load team roster
→ remove one player
→ add one player
→ generate valid default minutes
→ load versioned player projections
→ calculate team-impact difference
→ estimate projected win change
→ return assumptions and explanation factors
```

The interface may be visually rough. The end-to-end loop must work before supporting pages are polished.
